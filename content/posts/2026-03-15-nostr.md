# 聊聊 nostr 和 审查

## 引子：一场“下架”引发的集体追问

2023 年初，一款名为 **Damus** 的 iOS 社交应用在苹果 App Store 上线，仅一天后便从中国区商店悄然消失。官方公告语焉不详，仅称“因违反中国法律法规”，但用户社区迅速达成共识：这并非偶然的技术合规问题，而是一次对新型去中心化通信协议——**nostr**（Notes and Other Stuff Relayed）——的首次现实尺度审查响应。

有趣的是，Damus 本身并不托管任何内容；它不运营服务器，不存储用户消息，甚至不验证用户身份。它只是一个轻量客户端，连接到全球数百个公开、自愿运行的中继服务器（relays），通过私钥签名发布加密消息，并订阅他人广播的事件。它的核心逻辑完全运行在用户设备上，数据流经第三方中继，但中继本身无权解密、编辑或删除内容——除非主动选择不转发。

这种架构让 Damus 成为首个在主流移动平台大规模落地的、真正意义上“抗审查”的社交客户端。而它的下架，恰恰印证了其设计目标：不是规避审查，而是让审查在技术上变得低效、昂贵且不可持续。

本文将深入拆解 nostr 协议的本质机制，厘清它与传统中心化社交平台（如微博、Twitter/X）、乃至其他去中心化方案（如 Mastodon、ActivityPub）的根本差异；我们将逐层剖析其密码学基础、事件模型、中继协作逻辑、客户端行为规范；并通过可运行的 Python 与 JavaScript 示例，亲手构建一个最小可行的发布-订阅流程；最后，我们将直面最尖锐的问题：nostr 真的无法被审查吗？它的“抗审查性”是否存在边界？当国家力量、商业平台、网络基础设施与协议层发生碰撞时，技术理想主义能否守住底线？这场关于“谁控制消息、谁定义真相、谁承担后果”的博弈，远未结束——它刚刚进入第一回合。

本解读不预设立场，但坚持一个前提：理解技术，是参与公共讨论的第一步。只有看清协议如何工作，我们才能理性判断——它究竟在抵抗什么，又可能纵容什么。

## 一、协议溯源：nostr 不是“另一个 Twitter”，而是一张签名消息的广播网

要理解 nostr 的颠覆性，必须先放下“社交网络”的旧有范式。它既不是微博那样的信息流平台，也不是微信那样的熟人关系链工具，更不是 Mastodon 那样基于服务器联邦的分布式微博客系统。nostr 的原始定位极为朴素：**一个极简的、基于公钥密码学的、不可篡改的消息广播协议**。

它的诞生背景极具象征意义：2020 年底，Twitter 创始人 Jack Dorsey 在推特上发起一项实验——“如果我们要重建一个开放的社交网络，它应该有多小？” 他提出一个核心思想：**身份即公钥，消息即签名，分发靠中继，验证靠数学**。这一想法由开发者 [Globo](https://github.com/globee) 迅速实现为一份仅 300 行的原始规范草案，随后演变为今日的 [nostr-protocol](https://github.com/nostr-protocol) 开源标准。

### 1.1 核心哲学：三原则定义协议边界

nostr 严格遵循三条设计铁律，它们共同划定了协议的能力边界与安全假设：

1. **身份即公钥（Identity is Public Key）**  
   用户没有用户名、邮箱、手机号或任何中心化注册凭证。唯一标识是一个 32 字节的比特币风格 Secp256k1 公钥（以 `npub` 编码为 Base32 字符串）。私钥永远离线保存于用户设备，所有操作均由私钥签名完成。这意味着：创建账户零成本、零信任、零依赖；注销账户即销毁私钥——彻底、不可逆、无需通知任何人。

2. **消息即事件（Message is Event）**  
   所有交互被抽象为统一的数据结构——`Event`。它不是一个富文本帖子，而是一个严格定义的 JSON 对象，包含固定字段：`id`（事件 SHA256 哈希）、`pubkey`（发布者公钥）、`created_at`（Unix 时间戳秒级）、`kind`（消息类型编号）、`tags`（元数据标签数组）、`content`（任意 UTF-8 字符串，可为空）。`id` 由整个事件对象（除签名外）序列化后哈希生成，确保内容不可篡改。

3. **分发靠中继，验证靠数学（Relay for Distribution, Math for Verification）**  
   没有“官方服务器”。消息通过用户自主选择的中继服务器（如 `wss://nos.lol`、`wss://relay.damus.io`）广播。中继只做两件事：接收已签名事件、按规则存储与转发。它**不验证事件内容合法性，不检查用户是否“可信”，不执行任何业务逻辑**。验证完全由客户端完成：收到事件后，客户端用 `pubkey` 解析出公钥，用该公钥验证 `sig` 签名是否匹配 `id`，仅此而已。数学正确，即事件真实；数学错误，即伪造，直接丢弃。

这三点共同构成一个“零信任分发环”：中继不信任用户，用户不信任中继，客户端只信任密码学。

### 1.2 与主流方案的本质对比：一张表看懂差异

| 维度             | Twitter / 微博                         | Mastodon（ActivityPub）               | nostr                                     |
|------------------|----------------------------------------|----------------------------------------|-------------------------------------------|
| **身份模型**     | 中心化账号（用户名+密码/手机）         | 服务器账号（username@server.tld）      | 公钥即身份（`npub1...`），无注册过程     |
| **数据所有权**   | 平台完全拥有并控制全部数据             | 数据物理存储于各实例服务器，但协议绑定 | 数据由用户私钥签名，逻辑归属用户         |
| **分发机制**     | 中心化推送 + 缓存                      | 联邦式推送（服务器间 HTTP POST 同步）  | 广播式中继（客户端直连多个 wss 中继）    |
| **审查能力**     | 平台可实时删除、限流、屏蔽任意内容     | 实例管理员可封禁域、删帖、限制可见范围 | 中继可拒绝转发某事件，但无法阻止其被其他中继传播 |
| **离线能力**     | 强依赖在线，无本地数据持久化           | 客户端缓存有限，依赖服务器状态         | 客户端可完全离线生成事件，联网后批量广播 |
| **扩展性瓶颈**   | 中心化数据库与带宽                     | 联邦同步延迟与跨域认证开销             | 中继带宽与存储，但无状态、易横向扩展     |
| **信任假设**     | 用户信任平台不作恶                     | 用户信任所选实例管理员                 | 用户只信任密码学（ECDSA 签名）与自身私钥 |

关键洞察在于：Mastodon 的“去中心化”本质是**服务器治理去中心化**，而 nostr 的去中心化是**协议层与身份层的彻底解耦**。前者仍需你选择一个“主子服务器”来托管你的数字存在；后者则让你成为网络中的一个独立信源节点，自由选择任何中继作为临时广播通道——就像用不同邮局寄同一封信，邮局无法篡改信封上的签名，也无法阻止你另选一家。

### 1.3 协议演进：从 0 到 1.3，稳定背后的克制

截至 2024 年中，nostr 规范已迭代至 v1.3（[NIP-01](https://github.com/nostr-protocol/nips/blob/master/01.md) 为核心），但其主干结构三年未变。这种稳定性源于刻意为之的“极简主义”：

- **无内置好友关系**：`follow` 行为由客户端解析 `kind:3`（联系人列表）事件实现，中继不感知；
- **无内置点赞/转发**：`kind:7`（反应）、`kind:6`（转发）均为客户端解释的约定，中继只存储原始 JSON；
- **无内置时间线排序**：客户端按 `created_at` 或自定义算法（如热度、作者权重）聚合事件；
- **无内置搜索索引**：全文检索需客户端或第三方服务自行构建，中继不提供 `/search` API。

这种“不做决定”的设计，使 nostr 成为一个**协议基座**（Protocol Primitive），而非功能完备的应用。它像 TCP/IP 之于互联网——不关心你传的是网页还是视频，只确保字节流按规则送达并可验证。所有上层语义（社交图谱、内容推荐、内容审核），均由客户端或第三方服务在协议之上叠加实现。

这也解释了 Damus 下架的深层逻辑：苹果审核的不是 nostr 协议（它只是 WebSocket 通信），而是 Damus 客户端**默认连接的中继列表**（含境外节点）、**默认启用的 kind 类型**（如 `kind:1` 文本帖、`kind:30023` 音频附件）、以及**其 UI 层对敏感话题的展示策略**。审查对象，从来不是协议本身，而是协议之上的具体实现与配置。

> ✅ 小结：nostr 的本质，是一张由公钥签名驱动的、事件化的、中继广播的消息网络。它用密码学取代信任，用广播取代同步，用客户端智能取代服务器逻辑。理解这一点，是穿透所有“抗审查”叙事迷雾的第一束光。

## 二、密码学基石：Secp256k1 签名如何构筑不可抵赖的身份

nostr 的一切可信性，根植于一个古老而坚固的密码学原语：**椭圆曲线数字签名算法（ECDSA）在 Secp256k1 曲线上的应用**。这不是一个可选项，而是协议强制要求的底层支柱。本节将剥开数学外壳，用可验证的代码演示：为何一个 `npub` 地址能唯一代表你，为何一条 `sig` 签名能证明消息绝非伪造。

### 2.1 Secp256k1：比特币同源的安全根基

Secp256k1 是 NIST 标准化的一条椭圆曲线，其参数定义为：
- 域大小 `p = 2²⁵⁶ − 2³² − 977`
- 基点 `G = (Gx, Gy)`，其中 `Gx`, `Gy` 为超大整数
- 曲线方程：`y² ≡ x³ + 7 (mod p)`

它的核心优势在于：
- **计算高效**：相比 RSA，同等安全强度下密钥更短（256 位 vs 3072 位），签名/验签速度更快；
- **工业验证**：被比特币、以太坊等万亿级价值系统长期使用，攻击面已被充分挖掘；
- **标准化友好**：BIP-32/39/44 等助记词标准可无缝迁移至 nostr 身份管理。

在 nostr 中，用户私钥 `sk` 是一个 256 位随机整数（`1 ≤ sk < n`，`n` 为曲线阶）；公钥 `pk` 是 `sk × G`（标量乘法结果），表现为一个 65 字节坐标点（`04` + `x` + `y`），再经 SHA256 + RIPEMD160 哈希 + Base32 编码，得到人类可读的 `npub` 地址（如 `npub18ntzgj2r2m3s5a8h6v7f9q1x4c5b6d7e8f9g0h1i2j3k4l5m6n7o8p9q0r`）。

### 2.2 事件签名：从明文到不可篡改的完整链条

一个 nostr 事件（`Event`）的签名流程如下：

1. **构造待签名数据**：取事件对象中除 `sig` 字段外的所有字段（`id`, `pubkey`, `created_at`, `kind`, `tags`, `content`），按 JSON 序列化规则（键名升序、无空格、字符串双引号）拼接为字符串；
2. **计算事件 ID**：对该字符串进行 SHA256 哈希，得到 32 字节 `id`；
3. **签名生成**：用私钥 `sk` 对 `id` 字节进行 ECDSA 签名，输出 `r`, `s` 两个大整数，拼接为 64 字节 `sig`（`r` 32 字节 + `s` 32 字节）；
4. **事件组装**：将 `id` 和 `sig` 加入事件对象，形成完整可广播事件。

关键点在于：`id` 由事件内容决定，`sig` 由 `id` 和私钥决定。因此，**任何对 `content`、`tags`、`created_at` 的修改，都会导致 `id` 改变，从而使原有 `sig` 验证失败**。数学上，这是单向函数与离散对数难题的双重保障。

### 2.3 动手实践：用 Python 构建一个完整签名/验证流程

以下代码使用 `secp256k1` 和 `hashlib` 库，完整复现 nostr 事件签名与验证逻辑。所有注释均为中文，严格遵循规范。

```python
# -*- coding: utf-8 -*-
"""
nostr 事件签名与验证演示
依赖库：pip install secp256k1
说明：本示例严格遵循 NIP-01 规范
- 使用 Secp256k1 曲线
- 事件 ID 为 SHA256(content + pubkey + created_at + kind + tags)
- 签名对 id 字节进行 ECDSA 签名
"""

import json
import hashlib
import secrets
from secp256k1 import PrivateKey, PublicKey

def serialize_event_for_id(event):
    """按 NIP-01 规范序列化事件（不含 sig 字段）用于计算 id"""
    # 注意：JSON 键必须按字母顺序排列！
    keys_ordered = ["content", "created_at", "id", "kind", "pubkey", "sig", "tags"]
    # 构造一个只含必要字段的 dict，排除 sig
    data = {
        "content": event.get("content", ""),
        "created_at": event["created_at"],
        "kind": event["kind"],
        "pubkey": event["pubkey"],
        "tags": event.get("tags", [])
    }
    # JSON 序列化：无空格、键名升序、字符串双引号
    return json.dumps(data, separators=(',', ':'), sort_keys=True)

def compute_event_id(event):
    """计算事件 id：SHA256(serialize_event_for_id)"""
    serialized = serialize_event_for_id(event)
    return hashlib.sha256(serialized.encode('utf-8')).digest()

def generate_keypair():
    """生成 Secp256k1 密钥对"""
    privkey = PrivateKey()
    pubkey = privkey.pubkey
    # 将公钥（65字节）转换为 nostr npub 格式（Base32）
    # 步骤：pubkey_bytes -> SHA256 -> RIPEMD160 -> Base32
    from base64 import b32encode
    import hashlib
    pk_bytes = pubkey.serialize(compressed=False)  # 65字节
    sha = hashlib.sha256(pk_bytes).digest()
    ripemd = hashlib.new('ripemd160', sha).digest()
    npub = b32encode(ripemd).decode('ascii').lower().replace('=', '')
    return privkey, pubkey, npub

def sign_event(event, privkey):
    """用私钥对事件签名"""
    event_id = compute_event_id(event)
    # ECDSA 签名（返回 r,s 元组）
    sig_der = privkey.ecdsa_sign(event_id, raw=True)
    # 转换为 64 字节 sig（r 和 s 各 32 字节，大端填充）
    sig_bytes = privkey.ecdsa_serialize(sig_der)
    # 注意：nostr 要求 sig 为 64 字节，此处需手动截取/填充
    # 实际生产环境应使用更精确的序列化，此处为简化演示
    if len(sig_bytes) < 64:
        # 补零（实际中应确保 DER 解析正确）
        sig_bytes = sig_bytes.rjust(64, b'\x00')
    elif len(sig_bytes) > 64:
        sig_bytes = sig_bytes[:64]
    return event_id, sig_bytes

def verify_event(event):
    """验证事件签名是否有效"""
    try:
        # 解析 pubkey 字符串为 bytes（需先 Base32 解码 npub）
        # 此处简化：假设 pubkey 已是 33 字节压缩公钥（实际 npub 需解码）
        # 生产环境应使用 https://github.com/fiatjaf/nostr-tools 的 utils.npub_decode
        import base64
        # 模拟：从 npub 解码出 pubkey（省略复杂步骤，聚焦验证逻辑）
        # 真实验证需：npub -> ripemd160(sha256(pubkey)) -> 反向查表？不，应直接用 pubkey 字段
        # 因此，event['pubkey'] 必须是 33 或 65 字节的原始公钥 bytes
        # 本示例中，我们直接使用 PublicKey 构造
        pk_bytes = bytes.fromhex(event["pubkey"])  # 假设 pubkey 是 hex 字符串
        pubkey = PublicKey(pk_bytes, True)
        event_id = bytes.fromhex(event["id"])
        sig_bytes = bytes.fromhex(event["sig"])
        
        # 尝试解析 sig_bytes 为 DER 格式
        # 简化：直接调用 verify 方法（需适配）
        # 实际中，应使用 secp256k1.ecdsa_deserialize_compact
        return pubkey.ecdsa_verify(event_id, sig_bytes, raw=True)
    except Exception as e:
        print(f"验证失败：{e}")
        return False

# === 主流程演示 ===
if __name__ == "__main__":
    print("=== nostr 事件签名与验证演示 ===\n")

    # 1. 生成密钥对
    privkey, pubkey_obj, npub = generate_keypair()
    pubkey_bytes = pubkey_obj.serialize(compressed=True)  # 33字节压缩公钥
    pubkey_hex = pubkey_bytes.hex()
    print(f"生成 npub 地址：{npub}")
    print(f"对应公钥（hex）：{pubkey_hex}")
    print()

    # 2. 构造一个测试事件
    test_event = {
        "content": "Hello, nostr! 这是一条测试消息。",
        "created_at": 1672531200,  # 2023-01-01 00:00:00 UTC
        "kind": 1,
        "pubkey": pubkey_hex,  # 注意：这里填入 hex 字符串，非 npub
        "tags": []
    }

    # 3. 计算 id 并签名
    event_id, sig_bytes = sign_event(test_event, privkey)
    test_event["id"] = event_id.hex()
    test_event["sig"] = sig_bytes.hex()

    print("构造的事件：")
    print(json.dumps(test_event, indent=2, ensure_ascii=False))
    print()

    # 4. 验证事件
    is_valid = verify_event(test_event)
    print(f"事件签名验证结果：{'✅ 有效' if is_valid else '❌ 无效'}")
    print()

    # 5. 篡改测试：修改 content 后验证失败
    tampered_event = test_event.copy()
    tampered_event["content"] = "Hello, nostr! 这是一条被篡改的消息。"
    # 重新计算 id（此时 id 已不匹配原 sig）
    tampered_id = compute_event_id(tampered_event)
    tampered_event["id"] = tampered_id.hex()
    # 但 sig 未更新，仍为原 sig
    is_tampered_valid = verify_event(tampered_event)
    print("篡改 content 后验证：")
    print(f"事件 id 已变更：{tampered_event['id'][:16]}...")
    print(f"验证结果：{'✅ 有效' if is_tampered_valid else '❌ 无效（因签名不匹配）'}")
```

## 三、核心机制解析：约束求解与模型生成

Z3 是一个高性能的 SMT（Satisfiability Modulo Theories）求解器，其核心能力在于自动判断一组逻辑约束是否可满足，并在可满足时构造出具体的赋值模型（model）。这一过程并非暴力枚举，而是融合了布尔求解（SAT）、线性算术、位向量、数组、未解释函数等多种理论的协同推理。

当用户调用 `solver.check()` 时，Z3 内部会经历以下关键步骤：
- **约束归一化**：将高级表达式（如 `x > 2*y + 1`）转换为标准理论公式，分解为布尔结构与各理论层的子公式；
- **理论组合**：通过 Nelson-Oppen 方法协调不同理论（例如整数算术与集合操作）之间的变量等价关系；
- **冲突驱动学习**：一旦发现某组假设导致矛盾，Z3 会生成“冲突引理”（conflict clause）并回溯，避免重复探索无效路径；
- **模型提取**：若判定为 `sat`，则从最终的简化约束图中反向推导出每个变量的具体取值，构成完整模型。

值得注意的是，Z3 不保证返回“最优解”，而只保证返回一个可行解；若需优化（如最小化目标函数），需使用 `Optimize` 类配合 `minimize()` 或 `maximize()` 方法。

## 四、典型应用场景与代码示例

### 场景一：软件验证中的路径可行性分析  
在静态分析中，Z3 可用于判断某条程序执行路径是否存在输入使其被触发。例如，分析如下 C 伪代码片段：

```python
# 假设输入为整数 x 和 y
if x > 0 and y < 10:
    if x * y == 42:
        trigger_bug()
```

对应 Z3 建模如下：

```python
from z3 import Int, Solver

x, y = Int('x'), Int('y')
s = Solver()

# 编码路径条件
s.add(x > 0)
s.add(y < 10)
s.add(x * y == 42)

if s.check() == sat:
    m = s.model()
    print(f"触发该路径的输入：x = {m[x]}, y = {m[y]}")  # 输出：x = 6, y = 7（或其他解）
else:
    print("该路径不可达")
```

### 场景二：密码学约束还原  
在 CTF 或协议逆向中，Z3 常用于求解异或（XOR）、移位、模运算混合的方程组。例如，已知 `a ^ b == 0x1234` 且 `(a << 3) & 0xFFFF == 0x5678`，可直接建模求解：

```python
a, b = BitVec('a', 16), BitVec('b', 16)  # 使用 16 位位向量
s = Solver()
s.add(a ^ b == 0x1234)
s.add((a << 3) & 0xFFFF == 0x5678)

if s.check() == sat:
    model = s.model()
    print(f"a = 0x{model[a].as_long():04x}, b = 0x{model[b].as_long():04x}")
```

> 注：此处使用 `BitVec` 而非 `Int`，是因为位运算在整数域上不具唯一语义，而位向量理论能精确建模底层操作。

## 五、性能调优与常见陷阱

### ✅ 推荐实践
- **优先使用具体类型**：对整数运算，若范围确定，选用 `BitVec(n)` 比 `Int()` 更高效；对实数精度要求不高时，可用 `Real()` 配合 `ToReal()` 转换。
- **分批添加约束**：避免一次性 `add()` 大量无关约束；可先构建子求解器验证局部一致性。
- **启用增量模式**：调用 `s.push()` / `s.pop()` 管理上下文，复用已学习的冲突引理，显著提升连续查询效率。

### ❌ 高频误区
- **混淆 `==` 与 `Equals()`**：在 Z3 表达式中，`x == y` 创建的是 Z3 的相等谓词（正确），但 `x is y` 或 `x == y` 在 Python 层比较对象身份（错误）。
- **忽略未声明变量的作用域**：`Int('x')` 每次调用都创建新变量，`x1 = Int('x'); x2 = Int('x')` 实际定义了两个独立变量，而非同一变量。
- **过度依赖 `solve()` 辅助函数**：它仅适用于简单表达式，复杂约束（含量化、递归定义）必须显式构造 `Solver` 并调用 `check()`。

## 六、总结：Z3 的定位与工程价值

Z3 不是一个通用编程语言，也不是替代人工推理的“黑箱AI”。它的真正价值在于——将领域专家对系统行为的**形式化断言**（如“此协议永不泄露密钥”“该调度算法总能在 100ms 内响应”），转化为机器可验证的逻辑约束，并以极高的自动化程度完成存在性证明或反例构造。

在现代软件开发中，Z3 已深度集成于 Microsoft Windows 内核验证、AWS IAM 策略分析、Rust 编译器借用检查器、以及各类智能合约审计工具链中。掌握其建模思想，不仅能提升安全分析能力，更能培养一种“可验证设计”的工程思维：在编码之前，先清晰定义“什么是对的”，再让机器帮我们确认它是否真的成立。
